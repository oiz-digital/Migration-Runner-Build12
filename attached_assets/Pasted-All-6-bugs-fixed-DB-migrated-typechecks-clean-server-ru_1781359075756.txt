Audit Complete — 6 Bugs Fixed
🔴 Critical (Financial / Security)
1. AI Trading: Free INR → USDT Conversion
Users could subscribe to a plan with INR, then cancel and receive USDT instead — effectively converting currency at zero cost. Root cause: the cancel handler always credited USDT regardless of what was paid.

Fix: Added fundingCoinId and fundingAmount columns to the ai_trading_subscriptions table (DB migrated). Subscribe now records the exact coin and amount paid. Cancel returns that exact coin and amount. Legacy rows without fundingCoinId fall back to the old USDT behavior.

2. KYC Missing on Crypto Withdrawals (Bicrypto compat route)
The /finance/withdraw/spot route in the Bicrypto compatibility layer had no KYC level check — a user with zero KYC could initiate crypto withdrawals. The main exchange routes already required KYC Level 1; this was a gap in the compat shim.

Fix: Added kycLevel < 2 check (consistent with the main routes) before the withdrawal is processed.

🟠 High (Data Integrity)
3. P2P Ledger — All 4 Entries Hardcoded "0" for balanceBefore/After
Every wallet ledger entry for P2P (escrow lock, release to buyer, cancel/refund) recorded balanceBefore: "0", balanceAfter: "0". This corrupted the user's ledger history page and broke audit trails.

Fix: Added a wallet SELECT before each escrow operation to snapshot the pre-op balance, then computed balanceAfter arithmetically. Seller debit at release now correctly tracks p2pLocked (which is what actually changes there); lock and refund track balance.

4. Duplicate Bank Accounts
No uniqueness check on the bank account POST endpoint — users could add the same account number multiple times, cluttering the INR withdrawal flow.

Fix: Added a SELECT-before-INSERT check; returns 409 Conflict if the account number already exists for that user.

🟡 Medium (Performance / Financial Accuracy)
5. Fee Settings — No Caching (DB hit on every match iteration)
loadFeeSettings() and loadVipTiers() ran a full DB query on every call. Since both are called in the matching engine's hot path (up to ~200 iterations per order), this was ~200 unnecessary DB round-trips per busy order.

Fix: Added a module-level in-memory TTL cache (30 seconds) for both functions. Fee/tier changes propagate within 30s. An invalidateFeeCache() export is available for admin routes to call after writing new fee config.

6. Referral Commission on GST-Inclusive Fee
The referral chain was paid commission on final.fee which includes 18% GST on top of the base exchange fee. Referrers were being overpaid by ~18% at the exchange's expense.

Fix: Back-calculates the base fee as grossFee / (1 + gstPercent/100) before passing it to the referral engine. Uses the cached getSpotFeeRates() call so no extra DB hit.
